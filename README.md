# SwiftDeploy

A declarative infrastructure automation CLI that generates Nginx and Docker Compose configurations from a single `manifest.yaml`, manages container lifecycles, and keeps your stack running.

> `manifest.yaml` is the single source of truth. Every config file is generated — nothing is handwritten.

---

## Project Structure

```
swiftdeploy-project/
├── manifest.yaml          # Single source of truth — edit only this
├── swiftdeploy            # CLI entrypoint (bash, executable)
├── Dockerfile             # Builds the API service image
├── app/
│   └── main.py            # Python HTTP service (or main.go)
├── templates/
│   ├── nginx.conf.tmpl    # Nginx config template
│   └── docker-compose.yml.tmpl  # Docker Compose template
├── nginx.conf             # Generated by swiftdeploy init (do not edit)
├── docker-compose.yml     # Generated by swiftdeploy init (do not edit)
└── README.md
```

## Prerequisites

- Docker (with Compose plugin) installed and running
- `bash` 4+
- `yq` (YAML parser for bash) — install via:
  ```bash
  sudo wget -qO /usr/local/bin/yq https://github.com/mikefarah/yq/releases/latest/download/yq_linux_amd64
  sudo chmod +x /usr/local/bin/yq
  ```
- `nginx` (for config syntax validation via `nginx -t`)
  ```bash
  sudo apt install nginx -y   # Ubuntu/Debian
  ```

## manifest.yaml

The manifest describes your entire stack. Do not edit generated files — edit this instead.

```yaml
services:
  image: swift-deploy-1-node:latest
  port: 3000
  mode: stable           # "stable" or "canary"
  version: "1.0.0"

nginx:
  image: nginx:latest
  port: 8080
  proxy_timeout: 30s

network:
  name: swiftdeploy-net
  driver_type: bridge
```

## CLI Subcommands

All commands are run from the project root:

```bash
./swiftdeploy <subcommand> [flags]
```

---

### `init`

Reads `manifest.yaml` and generates `nginx.conf` and `docker-compose.yml` from templates.

```bash
./swiftdeploy init
```

**What it does:**
- Parses all fields from `manifest.yaml`
- Renders `templates/nginx.conf.tmpl` → `nginx.conf`
- Renders `templates/docker-compose.yml.tmpl` → `docker-compose.yml`

**Example output:**
```
[init] Parsing manifest.yaml...
[init] Generating nginx.conf from template...
[init] Generating docker-compose.yml from template...
[init] Done. Generated files are ready.
```

---

### `validate`

Runs 5 pre-flight checks and exits non-zero if any fail.

```bash
./swiftdeploy validate
```

**Checks performed:**
| # | Check | Pass | Fail |
|---|-------|------|------|
| 1 | `manifest.yaml` exists and is valid YAML | ✅ PASS | ❌ FAIL |
| 2 | All required fields are present and non-empty | ✅ PASS | ❌ FAIL |
| 3 | Docker image referenced in manifest exists locally | ✅ PASS | ❌ FAIL |
| 4 | Nginx port is not already bound on the host | ✅ PASS | ❌ FAIL |
| 5 | Generated `nginx.conf` is syntactically valid | ✅ PASS | ❌ FAIL |

**Example output:**
```
[validate] Check 1: manifest.yaml exists and is valid YAML ✅ PASS
[validate] Check 2: Required fields present ✅ PASS
[validate] Check 3: Docker image swift-deploy-1-node:latest found locally ✅ PASS
[validate] Check 4: Port 8080 is available ✅ PASS
[validate] Check 5: nginx.conf syntax is valid ✅ PASS
[validate] All checks passed. Ready to deploy.
```

---

### `deploy`

Runs `init`, brings up the full stack, and waits for health checks to pass (60s timeout).

```bash
./swiftdeploy deploy
```

**What it does:**
1. Calls `init` to regenerate configs
2. Runs `docker compose up -d`
3. Polls `GET /healthz` every 2 seconds until it returns 200 or 60s elapses
4. Reports success or timeout

**Example output:**
```
[deploy] Running init...
[deploy] Starting stack with docker compose...
[deploy] Waiting for health checks...
[deploy] ✅ Service healthy after 6s. Stack is live on port 8080.
```

---

### `promote`

Switches the deployment mode between `stable` and `canary` with a rolling restart of the service container only.

```bash
./swiftdeploy promote canary
./swiftdeploy promote stable
```

**What it does:**
1. Updates `mode` field in `manifest.yaml` in-place
2. Regenerates `docker-compose.yml` with the new `MODE` env var
3. Restarts the service container only (Nginx is untouched)
4. Confirms the new mode by hitting `/healthz`

**Example output — promoting to canary:**
```
[promote] Updating manifest.yaml mode → canary
[promote] Regenerating docker-compose.yml...
[promote] Restarting service container...
[promote] Confirming mode via /healthz...
[promote] ✅ Mode confirmed: canary. X-Mode: canary header active.
```

**Example output — reverting to stable:**
```
[promote] Updating manifest.yaml mode → stable
[promote] Regenerating docker-compose.yml...
[promote] Restarting service container...
[promote] Confirming mode via /healthz...
[promote] ✅ Mode confirmed: stable.
```

---

### `teardown`

Stops and removes all containers, networks, and volumes.

```bash
./swiftdeploy teardown
```

With `--clean` flag, also deletes generated config files:

```bash
./swiftdeploy teardown --clean
```

**Example output:**
```
[teardown] Stopping containers...
[teardown] Removing networks...
[teardown] Removing volumes...
[teardown] ✅ Stack torn down.

# With --clean:
[teardown] Removing generated nginx.conf and docker-compose.yml...
[teardown] ✅ Clean teardown complete.
```

## API Endpoints

The service runs internally on `services.port` (3000) and is only reachable through Nginx on `nginx.port` (8080).

### `GET /`

Returns a welcome message with current mode, version, and server timestamp.

```bash
curl http://localhost:8080/
```

```json
{
  "message": "SwiftDeploy service is running",
  "mode": "stable",
  "version": "1.0.0",
  "timestamp": "2026-05-02T14:30:00Z"
}
```

---

### `GET /healthz`

Returns liveness status and process uptime in seconds.

```bash
curl http://localhost:8080/healthz
```

```json
{
  "status": "ok",
  "uptime_seconds": 142
}
```

---

### `POST /chaos` *(canary mode only)*

Simulates degraded behaviour. Three modes:

**Slow mode** — sleep N seconds before responding:
```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "slow", "duration": 8}'
```

**Error mode** — return 500 on ~50% of subsequent requests:
```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "error", "rate": 0.5}'
```

**Recover** — cancel all active chaos:
```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "recover"}'
```

> ⚠️ `/chaos` returns `403 Forbidden` in stable mode.

## Nginx Behaviour

- Listens on `nginx.port` (8080)
- All traffic proxied to the service — service port is never exposed directly
- Timeouts set from `nginx.proxy_timeout`
- Custom JSON error bodies for 502, 503, 504:
  ```json
  {
    "error": "Bad Gateway",
    "code": 502,
    "service": "swift-deploy-1-node",
    "contact": "ops@swiftdeploy.local"
  }
  ```
- Response headers:
  - `X-Deployed-By: swiftdeploy` on every response
  - `X-Mode` forwarded from upstream (active in canary mode)
- Access log format:
  ```
  2026-05-02T14:30:00+00:00 | 200 | 0.003s | 172.18.0.3:3000 | GET / HTTP/1.1
  ```

---

## Security Practices

- Service container runs as a **non-root user**
- Dropped Linux capabilities (`cap_drop: ALL`)
- Service port is **never** published to the host — only Nginx is exposed
- Named volume mounted for logs
- Health check defined on `/healthz` in Docker Compose

## Full Workflow Example

```bash
# 1. Build the image
docker build -t swift-deploy-1-node:latest .

# 2. Validate the manifest and environment
./swiftdeploy validate

# 3. Deploy the stack
./swiftdeploy deploy

# 4. Test stable mode
curl http://localhost:8080/
curl http://localhost:8080/healthz

# 5. Promote to canary
./swiftdeploy promote canary

# 6. Trigger chaos
curl -X POST http://localhost:8080/chaos -H "Content-Type: application/json" -d '{"mode":"error","rate":0.5}'

# 7. Recover
curl -X POST http://localhost:8080/chaos -H "Content-Type: application/json" -d '{"mode":"recover"}'

# 8. Revert to stable
./swiftdeploy promote stable

# 9. Tear down
./swiftdeploy teardown --clean
```

## Notes

- Never edit `nginx.conf` or `docker-compose.yml` directly — they are overwritten on every `init`
- The grader will delete generated files and re-run `swiftdeploy init` to verify regeneration
- `manifest.yaml` is the **only** file you should modify manually
