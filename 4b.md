Overview

Hi Cool Keeds!

In Stage 4a, you built the engine. For part 2, you provide the "Eyes" (Metrics) and the "Brain" (Policy). Your swiftdeploy tool will now refuse to deploy or promote if the environment doesn't meet specific safety standards defined by policy.

1. Instrumentation (/metrics)
Your API service must now expose a /metrics endpoint in Prometheus text format. You are responsible for tracking:

Throughput & Errors: http_requests_total (labels: method, path, status_code).
Latency: http_request_duration_seconds (histogram with standard buckets).
State: app_uptime_seconds, app_mode (0=stable, 1=canary), and chaos_active (0=none, 1=slow, 2=error).


2. The Policy Sidecar (OPA)
Update your tool to include an Open Policy Agent (OPA) container in the generated docker-compose.yml.

Connectivity: The OPA container must be reachable by the swiftdeploy CLI but isolated from the public Nginx ingress.
The Core Principle: The CLI must not make any allow/deny decision itself. All decision logic lives exclusively in OPA.
Policy Source: swiftdeploy init must ensure OPA loads all .rego files from a policies/ directory. Threshold values and environment-specific limits must not be hardcoded inside Rego files.
Policy Structure: Organise policies by domain. Each domain owns exactly one question and one set of data it cares about. The CLI queries each domain independently and treats each answer as a separate signal i.e  a deploy could be blocked by one domain and approved by another. A change to one domain's policy must never require touching another.
Decision Input: Policy checks must be context-aware. A pre-deploy check and a pre-promote check are different questions and must reflect that in what data they send to OPA.
Decision Output: OPA must never return a bare boolean. Every decision must carry the reasoning behind it, and your CLI must surface that reasoning clearly to the operator.
Failure Handling: Your CLI must never crash or hang when OPA is unavailable. Every distinct failure mode must produce a different, human-readable outcome.


3. Enhanced CLI Capabilities
The swiftdeploy script gains three major functions:
A. Gated Lifecycle (The "Brain")
Before executing, deploying or promoting, the CLI must query OPA.

pre-deploy: Send host stats (Disk, CPU, Mem) to OPA. Block if policies fail.
pre-promote: Scrape /metrics, calculate Error Rate and P99 Latency, and send them to OPA. Block promotion if the Canary is "unhealthy.".


B. swiftdeploy status (The "Eyes")
A live-refreshing terminal dashboard that:

Scrapes /metrics and calculates real-time req/s and P99 latency.
Displays a "Policy Compliance" list showing which Rego rules are currently passing/failing.
Appends every scrape to a history.jsonl file for the audit trail.


C. swiftdeploy audit (The "Memory")
Generates an audit_report.md by parsing history.jsonl.

Timeline: A table showing a summary of when modes changed and when chaos was injected.
Violations: A dedicated section listing any time a policy check was triggered


4. Policy Requirements (Rego)
You must write at least two distinct Rego policies:

Infrastructure Policy: Deny deployment if Disk Free < 10GB or CPU Load > 2.0.
Canary Safety Policy: Deny promotion if Error Rate > 1% or P99 Latency > 500ms over the last 30 seconds.
